# QA Test Task ‚Äî Submission

End-to-end testing of the ad-tracking system (Android SDK app + writer/reader/config backend).
Found **5 bugs** (2 Critical), verified spec-compliant behaviours, and built a two-tier
automation suite (**28 automated checks**).

## Approach

The challenge was framed around more **manual** testing of the Android app. But after running the
system and studying how it's built, I saw a better opportunity: to write **stable end-to-end tests
that exercise several layers at once** ‚Äî mobile frontend / mobile SDK / backend.

For the Android side I chose **Kaspresso**. The catch is that Kaspresso tests (and the test runner
itself) live **inside the emulator** and can't reach the host machine's files ‚Äî which is exactly
where the backend writes the logs I needed to verify against. I solved it with an **Appium-style
approach**: a small **agent runs on the host** and communicates with the tests running inside the
emulator, so every UI action can be checked against what the backend actually recorded. The agent
was fully generated by Claude ‚Äî details in [`../android-tests/README.md`](../android-tests/README.md).

The core **vtc / vti** ratios can't be isolated from the UI (the app pins the ad id to `ad-001`),
so those are covered by a **Playwright** suite directly against the backend. Full version in
`00-test-report.md`.

## Frameworks & why

- **Playwright + TypeScript** (`e2e/`) ‚Äî backend tier (gRPC writer/reader + REST config). It's
  the task's starter harness; tests isolate by **unique ad per test**, so they're fast,
  deterministic, and need no reset.
- **Kaspresso + UiAutomator** (`android-tests/`) ‚Äî Android UI tier. A convenient harness: it
  wraps `adb`, works with device logs/logcat, and adds a step DSL, flaky-safety, and Allure on
  top of UiAutomator. Here it drives the app **black-box** by text/content-desc, because Espresso
  can't instrument a foreign *release* APK (no source or signing key). **With access to the app
  source** we could go finer-grained ‚Äî Espresso-level view assertions and direct SDK hooks instead
  of black-box selectors.
- **Python `log-agent`** ‚Äî the instrumentation runs on the emulator, which can't read the host
  filesystem; the agent serves/clears `writer/reader/config` logs and runs reader reads over
  `10.0.2.2`.
- **Allure** ‚Äî reporting; Kaspresso steps become Allure steps, tests carry `@Issue`/`@Description`.
- **Docker Compose** ‚Äî the backend stack.

## How to run

**Backend** (required for all tests):
```sh
cd backend && ./images/load.sh && docker compose up -d      # ./reset.sh to wipe state
```

**Backend tests** (Playwright ‚Äî no emulator needed):
```sh
cd e2e && npm install && npx playwright test                # 17 tests
```

**Android tests** (needs a running emulator ‚Äî Pixel 6 / API 33 recommended ‚Äî and JDK 17):
```sh
# one-time: point Gradle at your Android SDK (local.properties is machine-specific and not shipped)
cd android-tests
echo "sdk.dir=$HOME/Library/Android/sdk" > local.properties   # or export ANDROID_HOME=<sdk path>
./run-e2e.sh                                                   # full cycle: docker‚Üíinstall‚Üítest‚ÜíAllure
```
`run-e2e.sh` resets the backend, installs the APKs, runs the tests, generates + opens the
Allure report, and uninstalls afterwards (`NO_OPEN=1` to skip opening). If the paths at the top of
`run-e2e.sh` (adb / emulator / JDK 17) differ on your machine, adjust them there.

> **Note on artifacts:** generated outputs are not shipped ‚Äî `npm install` restores `e2e/node_modules`,
> and Gradle rebuilds `android-tests/app/build` on first run (the Gradle **wrapper** is included, so no
> Gradle install is needed).

## Notes

- **4 tests are intentional red guards** (BUG-1..4) ‚Äî via `test.fail()` (Playwright) and a real
  exit code (Android). They pass once the bug is fixed; that's the regression signal.
- **Isolation:** backend tests by unique ad, Android tests by session id + log-clear in teardown.
- The provided `e2e/tests/example.spec.ts` fails out of the box ‚Äî that failure **is** BUG-3.
